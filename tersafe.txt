libtersafe.so 环境检测逻辑汇总（按模块归类）

一、反调试 / 进程跟踪（ptrace / wait / signal）
sub_26254C：对目标 pid 执行 ptrace(PTRACE_ATTACH)；以是否成功作为“可被附加/是否已被其他调试器占用”的判断入口（控制流混淆，结果通过间接跳转分发）。
sub_262AAC：对目标 pid 执行 ptrace(PTRACE_DETACH)；用于在探测或跟踪流程结束后解除附加（控制流混淆，结果通过间接跳转分发）。
sub_262844：针对不同 request/事件分支执行 ptrace(PTRACE_CONT) 或 sub_262AAC（DETACH）；属于“被跟踪进程事件处理/继续运行”的调度逻辑（混淆较重）。
sub_262C7C：对 pid 发送 SIGSTOP(19) → waitpid → ptrace(PTRACE_DETACH) → SIGCONT(18)；用于“暂停-同步-解除跟踪-恢复”流程，常见于自跟踪/反调试管控。
sub_269C38：核心反调试/自跟踪入口之一：ptrace(PTRACE_ATTACH) 附加目标 pid，waitpid 等待其停止后使用 ptrace(PTRACE_SYSCALL) 驱动；失败或状态异常会写入状态码并 DETACH，体现“检测是否可附加/驱动系统调用跟踪”的能力。
sub_269484：对目标 pid 执行 ptrace(PTRACE_GETREGSET) 读取寄存器集，并检查 0xBEEF0000 魔数与 173 等字段；命中后可能触发 sub_269520 对寄存器进行修正，体现“通过寄存器/约定魔数进行通信或反调试控制”的逻辑。
sub_269520：对目标 pid 执行 ptrace(PTRACE_SETREGSET) 写回寄存器集；在 sub_269484 的条件满足时用于修改寄存器状态（可能用于跳过/修复/对抗外部跟踪）。
sub_269590：对目标 pid 执行 ptrace(PTRACE_SETREGSET, request=1028, …) 写入 4 字节数据；更像是设置某类调试/寄存器相关参数（具体含义取决于内核/架构实现）。
sub_269864：设置 *(ctx+116)=6（并伴随 pid/stat_loc/options/request 等注释痕迹）；属于 ptrace/wait 相关状态机的一环（反调试流程的内部状态推进，反编译不完整）。
sub_26B414：将字符串参数解析为 pid（atoi），尝试 ptrace(PTRACE_ATTACH)；若附加失败走错误分支，附加成功则 waitpid 并调用 sub_26B740 两次后 DETACH；属于“对指定 pid 的可附加性/状态读取”检测。
sub_26B740：与 sub_26B414 配套的混淆分发节点（根据 pid/参数决定后续分支）；用于反调试探测流程内部调度。
sub_262648：调用 sub_262AAC 的调度/包装（反编译不完整）；属于 ptrace 流程中的控制流拼接点。

二、系统调用完整性 / 父进程关联检查（syscall / getppid）
sub_269E78：syscall 包装器：当未开启某标志位时直接 syscall(sysno, …)；否则会缓存 getppid() 并发起 syscall(173, …, (sysno|0xBEEF0000), …) 的“带魔数调用”，若返回值与缓存的父进程 pid 相等且 sysno != 48879 则返回 -1。该结构体现“对系统调用链路进行一致性/反拦截检测”的意图（可用于发现被劫持的 syscall 路径或异常环境）。
sub_261988：调用 prctl(PR_SET_DUMPABLE=4, 1, 0, 0, 0)，并将结果写入 *(ctx+2166)；通常用于调整进程可被 ptrace/生成 core 的行为，配合自跟踪/反调试机制使用。
sub_26C0C0：调用 prctl(PR_SET_DUMPABLE=4, 1, 0, 0, 0) 后继续执行字符串/命令拼装与后续调用；体现“先调整 dumpable 再执行后续安全/检测流程”的初始化步骤（其余逻辑反编译不完整）。

sub_3FD4A0：inline hook 检测/处理的核心大函数（通过字符串可见特征包括：inline_hook_opcode_dismatch、set_inline_hook_error、ms_set_inlie_hook、ms_hook_opcode 等）；典型用途是校验目标函数入口指令序列是否被篡改、发现异常时记录错误并进入补救/上报流程（反编译输出过长且被截断，但从字符串与结构可确认其为 inline hook 检测模块）。
sub_28F32C：以 &ptrace 为输入，结合 sub_4EA6A8(31359) 的条件分支将 ptrace 指针交给某分发表处理；属于“对 ptrace 指针/调用路径进行校验或差异化处理”的一环（可视作反 hook 逻辑的组成部分）。

四、硬件/系统属性与能力检测（__system_property_get / getauxval / uname）
sub_5097C0：读取 getauxval(0x10) 并检查位 0x100；满足条件后读取系统属性 ro.arch 并与 “exynos9810” 比较，设置全局 byte_577FA8 标志。该逻辑用于“基于硬件能力 + SoC/架构标识进行环境分支/兼容性开关控制”。

五、文件系统/运行环境路径判断（Android 沙箱路径特征）
sub_273DC0：对路径 “/data/data” 执行存在性/可访问性检查（通过 sub_4B897C）；sub_4B897C 会先尝试内部文件系统查询，失败后回退到 opendir("/data/data")。该项不是简单字符串判断，而是确认 Android 典型应用沙箱根目录是否真实存在、是否可打开；可用于识别非 Android 宿主、裁剪环境、容器/模拟环境中的目录缺失或访问异常。

sub_3D7DDC：通过 getpid 与 sub_3D59FC 获取当前进程标识，遇到 ':' 时截断，再拼装 “/data/data/%s/files”；用于推导“当前进程理论上的标准应用私有目录”。该项可与实际运行路径/apk_path 对比，用于识别多开、插件化、容器重定向、非标准安装路径等。
sub_4CF084：与 sub_3D7DDC 类似，但目标路径为 “/data/user/10/%s/files”；用于推导多用户/特定 userId 下的应用私有目录。该项可用于识别 Android 多用户、工作资料夹、分身/双开、容器型虚拟环境中的路径差异。

sub_24F3EC：创建 socket(AF_INET=2, SOCK_DGRAM=2, IPPROTO_UDP=17)；用于验证网络栈可用性或为后续网络信息获取做准备（控制流混淆，结果通过间接跳转分发）。

八、已确认的环境结果字段 / 上报项（由结果对象读取并格式化输出）
sub_2BA300 附近代码块：依次从某结果对象中读取 apk_path / shell_ver / check_state / debugger / s_open_id / time_magic 等字段；每个字段都存在 “字段名”、“字段名:%s”、“字段名=%s” 三种格式串，说明这些不是普通调试日志，而是实际参与汇总、格式化和上报的环境检测结果。
apk_path：用于记录 APK 实际路径；可用于识别重打包、非标准安装目录、容器/沙箱路径异常、多开环境中的路径重定向等。
shell_ver：用于记录 shell/root 环境版本或类型；说明检测逻辑并非只判断是否 root，还会细分 shell 特征。
check_state：综合状态字段；更像某一类检测结论、风险态或策略状态码。
debugger：调试器状态字段；结合前述 ptrace / wait / syscall 检测，说明库会将“是否存在调试器/调试状态类型”整理为可上报字符串。
s_open_id：某类 open/shell/环境状态标识字段；具体语义未完全还原，但可确认属于环境检测结果对象的一部分。
time_magic：时间相关检测字段；可能用于记录时间一致性、时间扰动、sleep/clock 异常、加速器/时间欺骗等环境迹象。
tp2_sdk_ioctl：存在 “|emulator_name=” 字符串拼接逻辑，说明明确支持“模拟器名称”结果字段，而不是仅通过硬件参数做隐式分支。
sub_4CC4F0：通用结果序列化器，使用格式串 “name=%s|rate=%.2f|is_root=%d”。当前已确认的唯一调用点是 sub_2846AC（anti-clicker 链），因此更稳妥的结论是：它负责把“命中项名称 + 百分制分数 + 一位布尔标志”打包成统一上报字符串；字段名虽然叫 is_root，但从 xref 不能直接断言该函数只服务于 root 模块。
sub_254120 附近代码块：存在字符串 mt2_no_root，并参与多个子检测结果的组合判定；这不是单一开关，而是一段“多规则 + 多状态位 + 采样次数”的 root/环境异常组合逻辑。具体来看：先通过 sub_4EB778(0xD01A) 解出 mt2_no_r，并调用 sub_22DA58 查询布尔结果；再通过 sub_4E84FC(0xBD2D) 解出 attunlock4 并查询；随后直接以明文 mt2_no_root 再查询一次。之后结合 sub_2BD8E0 返回对象上的状态位（偏移 0x50）、上下文位（偏移 0x18）、循环计数（与 “>3 次” 条件相关）、sub_2DB8E8 返回值，以及 sub_2870E4 返回对象偏移 0x48 处是否等于 2 等条件做 OR 式组合判定。一旦命中，就在 0x25423C 分支置位状态标志，并通过 sub_4E8DF0(0x7811) 解出的 “mem_trap” 作为事件名上报。因此 mt2_no_root 更像 root/内存异常/环境异常检测链中的内部规则标签、豁免条件或策略状态，而不是公开软件名。

sub_4EB778(0xD01A)：解密得到 “mt2_no_r”；结合 0x2541B8 处明文 “mt2_no_root”，可视为同一 root 相关标签/检测项的截断或配套字符串。在 sub_254120 附近逻辑中，该字符串被作为查询 key 使用，返回布尔结果参与后续组合判定。

十、可推断的目标环境、检测项目与可能针对的软件类型
1. 调试器 / 跟踪器：通过 ptrace(PTRACE_ATTACH/DETACH/SYSCALL/GETREGSET/SETREGSET)、waitpid、SIGSTOP/SIGCONT、带魔数 syscall 等逻辑，目标是发现外部调试器、trace 占用、寄存器级干预、syscall 链路拦截。
2. Hook / 注入框架：通过关键 libc/API 函数指针基线、inline_hook_opcode_dismatch、ms_hook_opcode 等逻辑，目标是识别 inline hook、PLT/GOT hook、入口指令篡改、调用路径重定向。
3. Root / Shell 环境：通过 is_root、shell_ver、shell_report、mt2_no_root、check_state 等字段，目标是识别 root 状态、shell 类型、shell 版本、root 豁免/白名单状态，而非仅检查 su 是否存在。
4. 模拟器 / 虚拟运行环境：通过 emulator_name、ro.arch、getauxval、uname、Android 私有目录路径与多用户路径等信息，目标是识别模拟器名称、硬件/SoC 偏差、容器/虚拟化路径特征。
5. 多开 / 沙箱 / 非标准安装环境：通过 apk_path、/data/data、/data/user/10/%s/files 等路径推导和存在性检查，目标是识别多用户、多开容器、沙箱映射、安装路径异常。
6. 网络环境：通过创建 UDP socket、setsockopt、执行 “ip route get 8.8.8.8 2>&1”，目标是识别网络栈可用性、默认路由、出网接口、受限网络或异常路由环境。
7. 时间 / 行为异常：通过 time_magic 字段与时间相关子流程，目标可能包括时间加速、sleep/clock 失真、行为回放或环境时间欺骗。
8. 辅助栈回溯 / 内存检测：通过 libtprt.so、unwind_xx_info_query、mem_trap 等字符串，可推断存在辅助的调用栈/内存异常检测模块。

十一、关于“检测了什么软件”的当前结论
已明确确认的模块/组件：libtprt.so（辅助检测模块）。
已确认的环境类别：调试器、hook/注入、root/shell、模拟器、路径/沙箱、多开/多用户、网络环境、时间异常、内存陷阱/栈回溯辅助检测。
当前未发现明确点名的公开软件名：Frida、Xposed、LSPosed、Substrate、Magisk、Zygisk、Riru、GameGuardian、BlueStacks、Nox、VMOS 等字符串在当前样本中未以明文或已解密字符串出现。
因此更合理的判断是：该库主要采用“行为特征 + 环境字段 + 辅助模块”的通用检测框架，而不是单纯依赖公开软件名黑名单。

十二、补充分析：check_state、rate 的评分口径，以及它与 root / anti-clicker 的关系
1. check_state 字段本身不是在格式化输出点现场打分，而是“结果对象中的状态字符串”。证据是 0x2BA410 - 0x2BA46C 这一段只做三件事：通过结果对象的虚函数表入口（LDR X8, [X19,#0x48] / BLR X8）按 key="check_state" 取值；再分别格式化为 "check_state:%s"、"check_state=%s"；最后写入汇总输出。当前样本里 “check_state” 明文只出现于这三处格式串/取值点，没有发现以该 key 为名的独立赋值函数，因此更合理的解释是：check_state 是其他检测流程预先写入结果容器的“状态枚举/状态字符串”，而不是这里现算的一个数字分数。
2. 需要修正的是：此前把 41539 / 41553 这组阈值临时归到了 root 链，但继续上追 xref 后可确认它们实际落在 sub_284308 / sub_2846AC 这条 anti-clicker 流程，而不是 mt2_no_root 那条 root 判定链。也就是说，`ano_det_cnt=3`、`ano_touch_thres=70` 这些默认门槛描述的是“触控异常告警”的升级条件，不是 root 告警的评分阈值。
3. rate 字段的最终合法区间仍然可以在 sub_4CC4F0 中落实为 0.00 ~ 100.00。该函数只接受非空 name、且 a4(float) 满足 0<=a4<=100，然后格式化成 "name=%s|rate=%.2f|is_root=%d"。因此 rate 的数值域是明确的，但它属于“统一报告格式中的百分制分数”，不能再直接等同于 root 专属分数。

21. mpmc.dat 的解析逻辑也已基本还原：
    - 文件先被整体读入内存；
    - 前 4 字节必须等于 magic `0x20250812`；
    - 接着读取一个 32 位 CRC32 字段；
    - 然后代码把 CRC 字段所在的 4 字节临时清零，再对整份 blob 跑标准 CRC32（sub_2E6188，初值 0xFFFFFFFF，末尾按位取反，查表实现）；
    - CRC 正确后再读取 32 位 entry_count；
    - 随后按每条 24 字节的结构循环解析：`<Q key0><Q key1><I value0><I value1>`；
    - 每条记录都会被送入 sub_43442C 插入容器，最后再由 sub_434568 安装到全局策略对象，供 sub_22DA58 这类查询函数按 key 取值。

23. 对当前工作目录里的真实样本 `ano_tmp/mpmc.dat` 再做本地解析后，可以确认它只有 36 字节，magic/CRC 都正确，但 `entry_count=1`，也就是只包含 1 条 `<Q key0><Q key1><I value0><I value1>` 记录。这说明本地样本里的 mpmc.dat 不是一份“完整 root 策略总表”，更像单条索引/种子记录或某个子模块专用策略块。
24. 也正因为 `ano_tmp/mpmc.dat` 当前只有 1 条不透明记录，root 链里真正参与判定的 `mt2_no_r`、`attunlock4`、`mt2_no_root`、`shell_report` 这些 key，暂时还不能直接从 mpmc.dat 明文反推出 value；从实现上更合理的判断是：sub_22DA58 所查询的 root/shell 策略值，要么来自其它 ano_tmp 数据文件，要么来自 mpmc.dat 间接索引到的更大策略容器，而不是这 36 字节文件本体。
25. 从 `ano_tmp` 目录文件名和体积看，当前最像 root/shell 相关策略库或结果缓存的文件有：
    - `ace_shell_db.dat`：40 字节，文件名最直观地指向 shell 数据库/缓存。
    - `ano_ksh.dat`：117 字节，名字里含 `ksh`，更像 shell / key shell / kernel shell 侧的缩写数据。
    - `atti_obv.dat`：172 字节，名字更像 attestation / observation 类观测结果缓存。
    - `ano_rdp.dat`：40 字节，名字更像 root detect policy / risk data profile 一类缩写文件。
   这些文件当前都不是明文文本，且在 SO 字符串表里没有直接出现对应文件名，所以只能先归类为“高疑似 root/shell 策略或缓存文件”，不能直接把某个文件和某个检测项一一对应。
26. `ace_shell_db.dat` 的文件格式已经能再前进一步：它是一个非常小的定长容器，而不像真正的“大型数据库”。当前样本可拆成：
    - `0x00-0x03 = 20 21 05 26`：更像日期式 magic/tag（可读作 `20210526`）。
    - `0x04-0x07 = 0x00001234`：固定标志位/版本位。
    - `0x08-0x0B = 0x00000018`：payload 长度 24 字节。
    - `0x0C-0x23`：24 字节载荷。
    - `0x24-0x27 = 0x00005678`：尾部哨兵值。
   因而 `ace_shell_db.dat` 更像“shell 侧单条摘要记录/种子记录/缓存条目”，而不是可直接枚举多项策略值的完整表；它支持 `shell_report`/`shell_ver` 这条链上存在持久化缓存，但还不足以单独还原 root 主判定。
27. `ano_ksh.dat` 的结构也已经能大致分层：
    - 开头 4 字节是 `07 04 23 20`，若按 little-endian 解释正好是 `0x20230407`，像一个日期式 magic。
    - 末尾 4 字节再次出现同样的 `07 04 23 20`，说明该文件存在“头尾重复 magic”。
    - 倒数第 8~5 字节是 `D4 3A 45 88`，很像独立的 CRC32/校验值。
    - 中间主体可分为“前缀块 + 64 字节核心 blob + 少量状态/填充字节 + 校验/尾标记”。
   也就是说，`ano_ksh.dat` 更像一份带校验的加密缓存容器，而不是简单 KV 文本。
28. `atti_obv.dat` 与 `ano_ksh.dat` 存在强关联。对比字节流后可以确认：
    - `atti_obv.dat` 在偏移 `0x40` 处包含一段 64 字节数据，与 `ano_ksh.dat` 偏移 `0x21` 开始的 64 字节块完全一致。
    - `atti_obv.dat` 的后半段还呈现出明显的“长度 + blob”布局：`0x00000040` 后接 64 字节块，`0x00000020` 后接 32 字节块，末尾再跟 `0x00000001 + 20241228` 风格的尾标记。
   这说明 `atti_obv.dat` 很可能不是独立规则表，而是把 `ano_ksh.dat` 的核心 shell/ksh 观测块重新封装进 attestation/observation 容器中。结合 root 链里的 `attunlock4` 命名，这进一步增强了“shell 观测结果与 attestation/解锁态规则是联动的”这一判断。
29. 因此，结合 `ano_tmp` 目录与代码路径，当前最稳妥的落地结论是：`mpmc.dat` 已明确属于策略入口之一，但并不能单独解释 root 判定；真正驱动 root 链的策略项，至少包括 `mt2_no_r`、`attunlock4`、`mt2_no_root`、`shell_report` 这 4 个可见 key，以及 1 个匿名常量规则 ID `0x78859728`，而这些值很可能分散保存在 ano_tmp 目录下的其它加密缓存文件中。现阶段能落实的是：`ace_shell_db.dat` 更像 shell 结果缓存条目，`ano_ksh.dat` 更像带 CRC 的 ksh/shell 观测容器，`atti_obv.dat` 则像把该观测块再次封装进 attestation 观测结果里的上层容器。

十三、结合 ano_tmp 目录可落实的 root 策略项
1. `mt2_no_r`：由 `sub_4EB778(0xD01A)` 解出，再经 `sub_22D7C0 + sub_22DA58` 查询布尔值。在 0x2541E4 处只要该位非 0 就会直接跳到命中分支，因此它是 root 链里优先级最高、最像“快速命中/快速豁免判定”的策略位。由于命中后不是简单返回，而是统一走 `mem_trap` 上报，所以更像“root/shell 高风险标记”而不是普通日志开关。
2. `attunlock4`：由 `sub_4E84FC(0xBD2D)` 解出，在 0x2541FC - 0x254204 这一段要和 `sub_2BD8E0()+0x50` 联动后才生效。也就是说，它不是单独为真就触发，而是“全局状态对象已经进入某个可疑态 + attunlock4 规则也为真”才升级为命中；从语义上更像“解锁态 / attestation 异常态 / 系统完整性异常态”类选项。
3. `mt2_no_root`：这是 root 链里最明确的策略 key，在 0x2541B8 以明文出现。它不会一次命中就立刻告警，而是还要求循环/采样轮次 `W21 > 3`，然后才在 0x254208 - 0x25420C 进入命中分支。说明它承担的是“稳定命中后再升级”的主开关，比 `mt2_no_r` 更像主 root 结论位。
4. `shell_report`：这项不直接参与 `is_root` 判定，而是出现在 `sub_2BA828` 里控制 shell 结果的附加输出通道。只有它为真时，代码才会继续结合 `/sdcard/sdk/enable.log` 做额外动作，并在 `shell_ver` 汇总后再落一份本地输出。因此它本质上是“shell 结果上报/诊断开关”，不是 root 主判定位。
5. 匿名规则 ID `0x78859728`：在 0x254220 - 0x254230 调用 `sub_23B98C` 时作为常量传入，返回值等于 2 就会触发命中。继续上追 `sub_23B98C / sub_23BA9C` 后可以确认：这不是“临时比较常量”，而是挂在对象 `+0x400` 规则树上的持久化槽位 ID。它与 `0x78859725`、`0x7885972A` 属于同一 ID 命名空间，其中 `0x78859725` 已明确存在“读取后再递增写回”的用法，因此 `0x78859728` 更像一个多值状态槽/枚举槽，而不是一次性布尔开关。
6. 这些策略项在 root 链里的角色可以概括为：
    - `mt2_no_r`：快速命中位。
    - `attunlock4`：需联动全局状态位的条件命中位。
    - `mt2_no_root`：带最少采样轮次门槛的主命中位。
    - `shell_report`：shell 结果输出/诊断控制位。
    - `0x78859728`：匿名多值状态槽；当前命中阈值是“查询结果 == 2”。
7. 换句话说，当前样本里的 root 检测并不是“列一串 su/ro.debuggable 开关逐个判断”，而是“几个策略 key + 几个全局状态位 + 一个匿名多值规则槽”组合起来决定是否上报 `mem_trap`。ano_tmp 目录更像这些策略值的落地存储位置，但仅靠当前已解析的 `mpmc.dat` 还不能把每个 key 的取值完整枚举出来。

十四、root / shell 检测链的函数级细化（本次补充）
1. sub_253F30：这是第一条独立的 `mem_trap` 扫描/上报链。它遍历 `ctx+0x88 .. ctx+0x90` 之间的候选项表，每项都通过 `ctx+0xA0` 的回调做检查，并把输出标志写到栈上的临时字节。只有当“回调返回 0、条目自身状态为 0、且输出字节 == 1”同时成立时，才把 `ctx+0x1C` 计数加一；前 5 次命中都会调用 sub_264D30 取报告对象，再把 `sub_4E8DF0(0x7811)` 解出的 `mem_trap` 传给 `loc_2671DC`。这条链没有直接出现 `mt2_no_root` 字符串，更像“通用内存陷阱/探针命中后立即上报”的前置筛查器。
2. 0x2540A4 - 0x254274：这是第二条、也是目前最像 root / shell 复合判定核心的代码块。它同样先对当前候选指针 `X24` 调一次 `ctx+0xA0` 回调；若 `sub_2BD8E0()+0x51` 与 `ctx+0x41` 两个字节都为真，还会额外调用 `ctx+0xC0` 的回调并传入模式值 4，说明“同一候选对象在特定全局状态下会被做二次检查/处置”。
3. 同一代码块会先把当前候选整理成调试摘要字符串 `ptr:%p,i:%d,r:%d,tick:%d,ec:%d,rc:%d`，其中至少包含：候选指针、循环索引、局部计数、tick、错误计数和累计命中数。随后再进入真正的 root 相关 key 查询。
4. sub_22D7C0 + sub_22DA58：这一对函数反复组合出现，作用更像“取当前策略/结果上下文句柄，再按 key 查询布尔状态”。在 root 链里它们被用来查询三类 key：
    - `sub_4EB778(0xD01A)` -> `mt2_no_r`
    - `sub_4E84FC(0xBD2D)` -> `attunlock4`
    - 明文 `mt2_no_root`
   这三项都不是公开 root 工具名，而更像内部规则标签、豁免位或策略态 key。
5. 0x2541D0 - 0x254238 这一段的 OR 判定可以细化成 6 个触发源，命中任一项都会跳到 `0x25423C`：
    - `mt2_no_r` 查询结果非 0，直接触发。
    - `sub_2DB8E8()` 的返回位与 `ctx+0x18==0` 这两个“保护条件”都不成立时触发；从用法看，`sub_2DB8E8` 更像刷新一次全局运行态/快照态，而不是文件检测。
    - `sub_2BD8E0()+0x50 != 0` 且 `attunlock4 == 1` 时触发；说明全局状态对象上的 byte[0x50] 与 `attunlock4` 规则是联动判定的。
    - `mt2_no_root == 1` 且循环/采样计数 `W21 > 3` 时触发；也就是说 `mt2_no_root` 不是“一次命中立刻告警”，而是带有最少采样轮次门槛。
    - `sub_2870E4()->[0x48] == 2` 时触发；该函数返回另一个全局单例对象，因此这里等价于“某个全局环境状态机进入状态 2 即视为可疑”。
    - `sub_23B98C(..., 0x78859728, 0)` 返回 2 时触发；这里传入的是规则树里的常量槽位 ID，而不是明文字符串，实质上是在查询某个持久化数值状态是否已经进入状态 2。
6. 一旦进入 `0x25423C`，代码会执行 `STRB W8, [X19,#0x40]` 把当前上下文的命中位设为 1；随后取 sub_264D30 返回的 96 字节报告对象，把 `sub_4E8DF0(0x7811)` 解出的 `mem_trap` 作为事件名传给 `loc_2671DC`。这里 `W7` 传的是 `W24 & 1`，也就是 `mt2_no_root` 的最低位，说明上报层确实把 `mt2_no_root` 当作其中一个核心布尔结果位使用。
7. sub_2BD8E0：惰性初始化并返回一个 88 字节左右的全局单例对象。root 链里实际读取了它的 `+0x50` 与 `+0x51` 两个字节：`+0x50` 参与 `attunlock4` 联动命中，`+0x51` 则决定是否执行那次额外的二次回调。因此它更像“root / shell / trap 共享的运行时状态对象”，而不是某个一次性检测函数。
8. sub_2DB8E8：内部只是 `sub_294CD4()` + `sub_297BD8()` 两步调用，但在 root 链里的用法是“刷新后立刻取返回位并参与 OR 判定”。这说明它扮演的是“快照同步/状态刷新器”，其最低 bit 被上层直接当成保护位或异常位使用。
9. sub_2870E4：只返回一个全局对象指针，root 链随后读取该对象 `+0x48` 的整型值并与 2 比较。也就是说，真正有语义的是对象字段 `state == 2`，而不是 sub_2870E4 自身。当前还无法把这个状态 2 精确命名为“已 root / 已 shell / 已解锁”，但它显然是 root 复合判定中的强命中条件。
10. sub_23B98C：继续逆向后可以把它从“黑箱查询”进一步细化为“规则树读接口”。证据有三点：
    - 该函数实际使用 `X0/X1/X2` 三个参数，而不是反编译表面看到的单参数。
    - `sub_23BA9C` 是它的配套写接口：会在 `a1+0x400` 这棵树上按 `key=X1` 查找节点，并把 `value=X2` 写到节点 `+0x20`；若节点不存在则调用 `sub_1DD76C` 新建节点。
    - `sub_20F06C` 里存在“先用 `sub_23B98C(..., 0x78859725, 0)` 读取，再用 `sub_23BA9C(..., 0x78859725, old+1)` 写回”的完整闭环，证明这套 ID 空间保存的是可变数值状态。
   因此，root 链里的 `sub_23B98C(..., 0x78859728, 0)` 应理解为“查询当前上下文规则树中 ID=0x78859728 的数值状态”；而不是普通字符串 hash 比较。由于上层把返回值 `==2` 当作命中条件，更合理的语义是：这个槽位记录的是一个离散枚举/计数态，状态 2 表示“某类 root/shell/完整性异常已经升级到可判定级别”。
11. sub_264D30：负责惰性申请并初始化 96 字节报告对象；sub_253F30 和 0x25423C 两条 `mem_trap` 链都会先调用它，再进入 `loc_2671DC`。它本身不做环境检测，但它是 root/mem_trap 结果上报的统一入口前置对象。
12. sub_2BA828：这是 `shell_ver` 旁边的一条 shell 辅助通道。它先用 sub_2BAD2C 解出 `shell_report` 和 `"%s/%s"`，再通过 `sub_22D7C0 + sub_22DA58` 查询 `shell_report` 是否开启；随后调用 `sub_4EC738(0x218C)` / `sub_4E6B40(0x2202)` 拼出 `/sdcard/sdk/enable.log`，并通过 sub_4B83E8 检查该文件。说明 shell 相关结果除了正常汇总外，还受“策略开关 + 本地 enable.log 标记文件”双重控制，像是本地调试/诊断通道。
13. 0x2BA378 - 0x2BA408：这里先从结果对象按 key=`shell_ver` 取字符串，格式化为 `shell_ver:%s` / `shell_ver=%s` 写入汇总；紧接着调用 sub_2BA828。若 sub_2BA828 返回 bit0=1，就会再次把 `shell_ver=%s` 格式化到栈上并交给 sub_1EF6F4。由此可见，`shell_ver` 的真正生成位置不在这段代码里，这里做的是“读取现成 shell 结果 + 按 shell_report 规则选择是否额外落一份本地输出”。
14. 额外补扫典型 root 黑名单明文后，当前样本里仍未见 `/system/xbin/su`、`/sbin/su`、`magisk`、`zygisk`、`test-keys`、`ro.debuggable`、`ro.secure` 这类清晰字符串命中。因此更可靠的结论是：本样本里的 root 检测重点不在“明文文件/属性黑名单”，而在“策略 key + 全局状态对象 + mem_trap 事件 + shell_report 辅助通道”这组复合判定链。
