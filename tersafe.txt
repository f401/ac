libtersafe.so 环境检测逻辑汇总（按模块归类）

一、反调试 / 进程跟踪（ptrace / wait / signal）
sub_26254C：对目标 pid 执行 ptrace(PTRACE_ATTACH)；以是否成功作为“可被附加/是否已被其他调试器占用”的判断入口（控制流混淆，结果通过间接跳转分发）。
sub_262AAC：对目标 pid 执行 ptrace(PTRACE_DETACH)；用于在探测或跟踪流程结束后解除附加（控制流混淆，结果通过间接跳转分发）。
sub_262844：针对不同 request/事件分支执行 ptrace(PTRACE_CONT) 或 sub_262AAC（DETACH）；属于“被跟踪进程事件处理/继续运行”的调度逻辑（混淆较重）。
sub_262C7C：对 pid 发送 SIGSTOP(19) → waitpid → ptrace(PTRACE_DETACH) → SIGCONT(18)；用于“暂停-同步-解除跟踪-恢复”流程，常见于自跟踪/反调试管控。
sub_269C38：核心反调试/自跟踪入口之一：ptrace(PTRACE_ATTACH) 附加目标 pid，waitpid 等待其停止后使用 ptrace(PTRACE_SYSCALL) 驱动；失败或状态异常会写入状态码并 DETACH，体现“检测是否可附加/驱动系统调用跟踪”的能力。
sub_269484：对目标 pid 执行 ptrace(PTRACE_GETREGSET) 读取寄存器集，并检查 0xBEEF0000 魔数与 173 等字段；命中后可能触发 sub_269520 对寄存器进行修正，体现“通过寄存器/约定魔数进行通信或反调试控制”的逻辑。
sub_269520：对目标 pid 执行 ptrace(PTRACE_SETREGSET) 写回寄存器集；在 sub_269484 的条件满足时用于修改寄存器状态（可能用于跳过/修复/对抗外部跟踪）。
sub_269590：对目标 pid 执行 ptrace(PTRACE_SETREGSET, request=1028, …) 写入 4 字节数据；更像是设置某类调试/寄存器相关参数（具体含义取决于内核/架构实现）。
sub_269864：设置 *(ctx+116)=6（并伴随 pid/stat_loc/options/request 等注释痕迹）；属于 ptrace/wait 相关状态机的一环（反调试流程的内部状态推进，反编译不完整）。
sub_26B414：将字符串参数解析为 pid（atoi），尝试 ptrace(PTRACE_ATTACH)；若附加失败走错误分支，附加成功则 waitpid 并调用 sub_26B740 两次后 DETACH；属于“对指定 pid 的可附加性/状态读取”检测。
sub_26B740：与 sub_26B414 配套的混淆分发节点（根据 pid/参数决定后续分支）；用于反调试探测流程内部调度。
sub_262648：调用 sub_262AAC 的调度/包装（反编译不完整）；属于 ptrace 流程中的控制流拼接点。

二、系统调用完整性 / 父进程关联检查（syscall / getppid）
sub_269E78：syscall 包装器：当未开启某标志位时直接 syscall(sysno, …)；否则会缓存 getppid() 并发起 syscall(173, …, (sysno|0xBEEF0000), …) 的“带魔数调用”，若返回值与缓存的父进程 pid 相等且 sysno != 48879 则返回 -1。该结构体现“对系统调用链路进行一致性/反拦截检测”的意图（可用于发现被劫持的 syscall 路径或异常环境）。
sub_261988：调用 prctl(PR_SET_DUMPABLE=4, 1, 0, 0, 0)，并将结果写入 *(ctx+2166)；通常用于调整进程可被 ptrace/生成 core 的行为，配合自跟踪/反调试机制使用。
sub_26C0C0：调用 prctl(PR_SET_DUMPABLE=4, 1, 0, 0, 0) 后继续执行字符串/命令拼装与后续调用；体现“先调整 dumpable 再执行后续安全/检测流程”的初始化步骤（其余逻辑反编译不完整）。

三、Hook / 篡改检测（函数指针基线 + inline hook）
sub_25B144：构建一份需要被关注的函数指针清单（open/read/write/fopen/fread/fgets/fseek/readdir/syscall/access/stat/fstat/getenv/kill/gettimeofday/sleep/select/connect/send/recv/opendir/ptrace/execl/clock_gettime/fdopen/getpid/dlopen/dlclose/pthread_create/fork/mmap，以及 tss_sdk_init 等）。这类清单通常用于“检测关键 API 是否被 inline hook/PLT hook/重定向”或用于后续一致性校验。
sub_25B464：将单个函数指针节点分配并插入到链表结构中（为 sub_25B144 的清单构建提供基础）。
sub_3FD4A0：inline hook 检测/处理的核心大函数（通过字符串可见特征包括：inline_hook_opcode_dismatch、set_inline_hook_error、ms_set_inlie_hook、ms_hook_opcode 等）；典型用途是校验目标函数入口指令序列是否被篡改、发现异常时记录错误并进入补救/上报流程（反编译输出过长且被截断，但从字符串与结构可确认其为 inline hook 检测模块）。
sub_28F32C：以 &ptrace 为输入，结合 sub_4EA6A8(31359) 的条件分支将 ptrace 指针交给某分发表处理；属于“对 ptrace 指针/调用路径进行校验或差异化处理”的一环（可视作反 hook 逻辑的组成部分）。

四、硬件/系统属性与能力检测（__system_property_get / getauxval / uname）
sub_5097C0：读取 getauxval(0x10) 并检查位 0x100；满足条件后读取系统属性 ro.arch 并与 “exynos9810” 比较，设置全局 byte_577FA8 标志。该逻辑用于“基于硬件能力 + SoC/架构标识进行环境分支/兼容性开关控制”。
sub_509DF4：一次性初始化逻辑：读取 ro.arch，若为空或不等于 “exynos9810”，则读取 getauxval(0x10) 与 getauxval(0x1A)，并将组合后的标志传入 sub_509858；用于“根据硬件能力/平台信息初始化某项底层能力或保护机制”（具体效果取决于 sub_509858）。
sub_2CC0F4：调用 uname() 获取系统信息，并将结果与解密得到的格式字符串通过 sub_4A54A4 拼装输出；用于“采集内核/系统版本信息”的环境信息采集逻辑（可能用于上报或策略分支）。

五、文件系统/运行环境路径判断（Android 沙箱路径特征）
sub_273DC0：对路径 “/data/data” 执行存在性/可访问性检查（通过 sub_4B897C）；sub_4B897C 会先尝试内部文件系统查询，失败后回退到 opendir("/data/data")。该项不是简单字符串判断，而是确认 Android 典型应用沙箱根目录是否真实存在、是否可打开；可用于识别非 Android 宿主、裁剪环境、容器/模拟环境中的目录缺失或访问异常。
sub_273F20：围绕 “/data/data” 的进一步分流逻辑。该函数先通过 sub_4A21F0 / sub_4A4688 做路径匹配，再调用 sub_4B8A1C 对目录句柄执行 readdir；随后在 0x274000 附近拼接 “/data/data/<entry>” 并对拼出的路径执行 stat。说明此处不仅检查 “/data/data” 是否存在，还会枚举其下目录项并校验目录结构/文件属性，可用于识别应用包目录枚举结果、异常挂载、目录映射或容器伪造路径。
sub_3D7DDC：通过 getpid 与 sub_3D59FC 获取当前进程标识，遇到 ':' 时截断，再拼装 “/data/data/%s/files”；用于推导“当前进程理论上的标准应用私有目录”。该项可与实际运行路径/apk_path 对比，用于识别多开、插件化、容器重定向、非标准安装路径等。
sub_4CF084：与 sub_3D7DDC 类似，但目标路径为 “/data/user/10/%s/files”；用于推导多用户/特定 userId 下的应用私有目录。该项可用于识别 Android 多用户、工作资料夹、分身/双开、容器型虚拟环境中的路径差异。

六、网络环境探测（socket + 外部命令获取路由）
sub_24F3EC：创建 socket(AF_INET=2, SOCK_DGRAM=2, IPPROTO_UDP=17)；用于验证网络栈可用性或为后续网络信息获取做准备（控制流混淆，结果通过间接跳转分发）。
sub_24F46C：对 fd 执行 setsockopt(fd, SOL_SOCKET=1, optname=2, optval=1)；典型为设置 SO_REUSEADDR 等 socket 选项，用于网络探测流程的 socket 初始化。
sub_24F984：通过 popen 执行外部命令并读取输出；命令字符串经解密后为：ip route get 8.8.8.8 2>&1，模式为：r。该逻辑用于获取默认路由/出接口等信息（可用于网络环境识别，例如出网接口、路由异常、受限网络等）。
sub_24EE38：网络探测流程的编排/状态机入口：根据多个开关位与参数，循环调用 sub_24F3EC 等子流程，最多尝试 5 次；用于对网络相关环境信息进行综合判断（反编译输出被截断，但从结构可确认其为 orchestrator）。

七、字符串解密支撑（用于隐藏环境检测的关键常量/命令/格式串）
sub_4E4A30：返回加密字符串表基址（&unk_FBBB0）。
sub_4E4A44：返回解密缓存区/状态基址（&unk_568FC4）。
sub_4E6498：字符串解密算法变体 A（每字节 key 以 +i+7 更新）；用于恢复如 popen 模式等明文（示例：offset=0 解密得到 “r”）。

八、已确认的环境结果字段 / 上报项（由结果对象读取并格式化输出）
sub_2BA300 附近代码块：依次从某结果对象中读取 apk_path / shell_ver / check_state / debugger / s_open_id / time_magic 等字段；每个字段都存在 “字段名”、“字段名:%s”、“字段名=%s” 三种格式串，说明这些不是普通调试日志，而是实际参与汇总、格式化和上报的环境检测结果。
apk_path：用于记录 APK 实际路径；可用于识别重打包、非标准安装目录、容器/沙箱路径异常、多开环境中的路径重定向等。
shell_ver：用于记录 shell/root 环境版本或类型；说明检测逻辑并非只判断是否 root，还会细分 shell 特征。
check_state：综合状态字段；更像某一类检测结论、风险态或策略状态码。
debugger：调试器状态字段；结合前述 ptrace / wait / syscall 检测，说明库会将“是否存在调试器/调试状态类型”整理为可上报字符串。
s_open_id：某类 open/shell/环境状态标识字段；具体语义未完全还原，但可确认属于环境检测结果对象的一部分。
time_magic：时间相关检测字段；可能用于记录时间一致性、时间扰动、sleep/clock 异常、加速器/时间欺骗等环境迹象。
tp2_sdk_ioctl：存在 “|emulator_name=” 字符串拼接逻辑，说明明确支持“模拟器名称”结果字段，而不是仅通过硬件参数做隐式分支。
sub_4CC4F0：使用格式串 “name=%s|rate=%.2f|is_root=%d”；说明 root 类检测结果会附带“命中项名称 + 风险评分 + 是否 root”的组合信息，而非单一布尔判断。
sub_254120 附近代码块：存在字符串 mt2_no_root，并参与多个子检测结果的组合判定；这不是单一开关，而是一段“多规则 + 多状态位 + 采样次数”的 root/环境异常组合逻辑。具体来看：先通过 sub_4EB778(0xD01A) 解出 mt2_no_r，并调用 sub_22DA58 查询布尔结果；再通过 sub_4E84FC(0xBD2D) 解出 attunlock4 并查询；随后直接以明文 mt2_no_root 再查询一次。之后结合 sub_2BD8E0 返回对象上的状态位（偏移 0x50）、上下文位（偏移 0x18）、循环计数（与 “>3 次” 条件相关）、sub_2DB8E8 返回值，以及 sub_2870E4 返回对象偏移 0x48 处是否等于 2 等条件做 OR 式组合判定。一旦命中，就在 0x25423C 分支置位状态标志，并通过 sub_4E8DF0(0x7811) 解出的 “mem_trap” 作为事件名上报。因此 mt2_no_root 更像 root/内存异常/环境异常检测链中的内部规则标签、豁免条件或策略状态，而不是公开软件名。

sub_4EB778(0xD01A)：解密得到 “mt2_no_r”；结合 0x2541B8 处明文 “mt2_no_root”，可视为同一 root 相关标签/检测项的截断或配套字符串。在 sub_254120 附近逻辑中，该字符串被作为查询 key 使用，返回布尔结果参与后续组合判定。

sub_4E8DF0(0x7811)：解密得到 “mem_trap”；高度疑似“内存陷阱/内存探针”类检测项，偏向反注入、反修改、反扫描。在 mt2_no_root 组合逻辑命中后，该字符串会作为上报/事件标签被传入后续处理函数。

sub_2BA76C - sub_2BA804：调用 dlopen("libtprt.so") 并 dlsym("unwind_xx_info_query")；属于加载辅助检测逻辑的明确证据。

sub_2BAD2C：对内置字节串执行逐字节异或恢复；可解出 “shell_report” 与 “%s/%s”等常量，说明 shell 检测存在单独的结果/上报通道。

十、可推断的目标环境、检测项目与可能针对的软件类型
1. 调试器 / 跟踪器：通过 ptrace(PTRACE_ATTACH/DETACH/SYSCALL/GETREGSET/SETREGSET)、waitpid、SIGSTOP/SIGCONT、带魔数 syscall 等逻辑，目标是发现外部调试器、trace 占用、寄存器级干预、syscall 链路拦截。
2. Hook / 注入框架：通过关键 libc/API 函数指针基线、inline_hook_opcode_dismatch、ms_hook_opcode 等逻辑，目标是识别 inline hook、PLT/GOT hook、入口指令篡改、调用路径重定向。
3. Root / Shell 环境：通过 is_root、shell_ver、shell_report、mt2_no_root、check_state 等字段，目标是识别 root 状态、shell 类型、shell 版本、root 豁免/白名单状态，而非仅检查 su 是否存在。
4. 模拟器 / 虚拟运行环境：通过 emulator_name、ro.arch、getauxval、uname、Android 私有目录路径与多用户路径等信息，目标是识别模拟器名称、硬件/SoC 偏差、容器/虚拟化路径特征。
5. 多开 / 沙箱 / 非标准安装环境：通过 apk_path、/data/data、/data/user/10/%s/files 等路径推导和存在性检查，目标是识别多用户、多开容器、沙箱映射、安装路径异常。
6. 网络环境：通过创建 UDP socket、setsockopt、执行 “ip route get 8.8.8.8 2>&1”，目标是识别网络栈可用性、默认路由、出网接口、受限网络或异常路由环境。
7. 时间 / 行为异常：通过 time_magic 字段与时间相关子流程，目标可能包括时间加速、sleep/clock 失真、行为回放或环境时间欺骗。
8. 辅助栈回溯 / 内存检测：通过 libtprt.so、unwind_xx_info_query、mem_trap 等字符串，可推断存在辅助的调用栈/内存异常检测模块。

十一、关于“检测了什么软件”的当前结论
已明确确认的模块/组件：libtprt.so（辅助检测模块）。
已确认的环境类别：调试器、hook/注入、root/shell、模拟器、路径/沙箱、多开/多用户、网络环境、时间异常、内存陷阱/栈回溯辅助检测。
当前未发现明确点名的公开软件名：Frida、Xposed、LSPosed、Substrate、Magisk、Zygisk、Riru、GameGuardian、BlueStacks、Nox、VMOS 等字符串在当前样本中未以明文或已解密字符串出现。
因此更合理的判断是：该库主要采用“行为特征 + 环境字段 + 辅助模块”的通用检测框架，而不是单纯依赖公开软件名黑名单。
